# 10 Gaps

## What exists and works

- Next.js 16.2.2 + React 19 + TS app under `training-hub/`, deployed to Vercel
- Supabase project `EVC` (`xkfvipcxnzwyskknkmpj`), Postgres 17, healthy
- Core data model is in place: `employees`, `training_types`, `training_records`, `excusals`, `auto_fill_rules`, `enrollments`, `training_sessions`
- `training_records_emp_type_date_unique` (employee_id, training_type_id, completion_date) prevents dupes
- `apply_auto_fill` trigger has the recursion fix and bails when source = 'auto_fill'
- `calculate_expiration` trigger fires correctly: 0 records have null expiry where renewal_years > 0
- `employees_name_unique_ci` functional index exists, so the upsert RPCs that target it actually work
- `upsert_employees_from_sheet` is the two pass version: by `employee_number` first, then by lower-cased name
- `aliases TEXT[]` column on employees with GIN index
- `employee_compliance` view computes compliance status from completions, expirations, excusals
- 16 client side pages and 40 API routes already wired up against Supabase via `lib/training-data.ts`, `lib/hub-settings.ts`
- 387 active employees in the live DB
- 1655 training records loaded across `merged_sheet`, `auto_fill`, `training_records_sheet`

## What exists but looks half built or broken

- Two identical functional unique indexes on employees: `employees_name_unique_ci` and `idx_employees_unique_name`
- Migration source folder is out of sync with live DB: 17 live migrations vs 12 repo files; at least 4 changes only exist on the live DB
- `lib/exclude-list.ts` and `lib/capacity-overrides.ts` are legacy file backed helpers, superseded by `lib/hub-settings.ts`, still in tree
- `components/ui/DataState.tsx` default loading message says "Loading data from Google Sheets..."
- `training_rules` table exists but is empty; all dept training rules live in `hub_settings` as pipe delimited strings instead
- `training_sessions`, `enrollments`, `training_schedules`, `notifications`, `removal_log`, `archived_sessions` are all empty (the scheduling/audit features were never used)
- Only 2 of 39 `training_types` have `renewal_years > 0` (CPR/FA, Med Recert); only CPR/FA has `is_required=true` at the type level
- `training_types.column_key` is not unique, so any future `ON CONFLICT (column_key)` upsert will error
- `excusals` table has 11,562 rows for 387 active employees, 11,407 of them auto generated by the merged sheet sync; almost certainly noise
- 1838 inactive employees and 1680 with null `employee_number` are bloating the table
- `training_records.pass_fail`, `arrival_time`, `end_time`, etc. are all TEXT, no validation, never enforced
- Repo migration 003 (`003_seed_data.sql`) uses an `ON CONFLICT (column_key)` clause that has no matching index; it was never actually applied to live DB
- The `006` prefix collides between two repo files; the live DB simply has different names
- `data-health` page and API exist, suggesting Kyle already knew the DB was drifty
- Sources for training_records use 3 distinct strings (`merged_sheet`, `auto_fill`, `training_records_sheet`) — none of them say `paylocity`, `phs`, `access`, or `signin`

## What's missing vs the target model and features

- No table for `training_aliases` per source (the existing `training_aliases` table is keyed only on training_type, not on source system)
- No `imports` / run log table (only `hub_settings` rows of `type='sync_log'`, which is a hack)
- No `unresolved_people` review table
- No `unknown_trainings` review table
- No canonical `paylocity_id` field on employees (the existing `employee_number` is close but null on 75% of rows)
- No required-trainings join table that's actually populated; current dept rules live as pipe delimited strings in `hub_settings`
- Compliance view does not separate `expired` vs `due_soon` by the brief's 30 day default (currently uses 60 days)
- No "master_completions" view computing winning completion per employee per training across sources
- No public sign in page (target feature 7.2). Old Google Form is still live, no in app replacement
- No imports page UI (target feature 7.3). No Paylocity CSV ingest, no PHS CSV ingest, no preview-then-commit flow
- No resolution review UI (target feature 7.4)
- No employee detail UI route (target 7.6) — `/api/employee-detail` exists, no page wraps it
- No training detail UI route (target 7.7)
- No run log UI page beyond `/sync` (which wraps the `sync_log` hub_settings hack)
- No Supabase Auth roles (HR admin vs viewer); current login uses a shared HR_PASSWORD env var
- No CSV export on the compliance dashboard
- No filter by location anywhere (location is not even a column on `employees`)
- No `program` data populated (every recent employee row has `program=null`)
- No generated TypeScript types from Supabase (`types/database.ts` is hand written)
- No tests, no test runner installed
